# Blink — Image System Configuration

## Overview

Every article gets a **character-driven AI-generated scene** as the hero image.
The character is Blink's builder hero — a consistent clay 3D developer figure who appears
in every article wearing a different costume suited to the topic.

> ⚠️ **NO dark gradient.** The hero image is used directly — the blog page shows the title
> separately; social/OG image is code-generated by `opengraph-image.tsx`.
> **NEVER** add "left side darker" or "left third darker" to the prompt — it bakes a dark
> overlay into the image that looks broken in card thumbnails. Use centered, full-frame always.

## CHARACTER ANCHOR (copy exactly — never modify)

```
Same clay 3D character: young developer, brown medium-length hair, big expressive eyes,
3D clay animation render style soft matte tactile finish, wearing their signature bright
electric-blue hoodie with white lightning bolt logo on chest.
```

This anchor is **prepended to every scene prompt, verbatim.** The character's face, hair,
and blue lightning hoodie must be visible in every image. Costumes layer ON TOP of the
hoodie (detective coat, space suit, blazer, hood-up) — the hoodie always shows through.

## Image Generation Call

```javascript
generate_image(
  prompt: "[CHARACTER ANCHOR] NEW SCENE: [scene description]. Expression: [expression].
           Background: [environment]. Centered composition, full frame. No dark gradients.
           No text. 16:9 wide blog header composition.",
  aspect_ratio: "16:9",
  output_format: "webp"
)
```

**Hard rules:**
- Always `aspect_ratio: "16:9"`, always `output_format: "webp"`
- Always use centered, full-frame composition (no dark left gradient — looks broken in thumbnails)
- Never include text in the image — the blog page shows the title separately
- The blue lightning hoodie must be visible in every image (even partially under a costume)
- The CHARACTER ANCHOR must be copied verbatim — never paraphrase it
- Replace every `[bracketed]` placeholder with article-specific content before calling

## CDN Upload

After generating the hero image, re-host on Blink CDN immediately:

```bash
cms_upload_asset(
  url: "[URL returned by generate_image — will look like cdn.blink.new/ai-generated/...]",
  filename: "[slug]-hero.webp",
  alt_text: "[Article Title]"
)
# Returns: { "public_url": "https://cdn.blink.new/cms/mcp-uploads/[slug]-hero.webp" }
```

> ⚠️ **`ai-generated/` URLs are EPHEMERAL — they expire within days.**
> `generate_image` returns `cdn.blink.new/ai-generated/[id].webp`. This looks like a
> permanent CDN URL but it is NOT — it is temporary storage that gets cleaned up.
> You MUST call `cms_upload_asset` to re-host to `cms/mcp-uploads/` even though the URL
> already appears to be on cdn.blink.new. Only `cms/mcp-uploads/` URLs are permanent.
> **NEVER embed a `cdn.blink.new/ai-generated/` URL directly in article content.**
>
> ⚠️ **Image URL Masking:** MCP outputs may show `[REDACTED]` for the domain. This is a
> display mask only. Real domain is always `https://cdn.blink.new`.
> **NEVER store `[REDACTED]` literally.** Always reconstruct: `https://cdn.blink.new` + path.

## Step 2c — Inline Images (2-3 per article)

### Which type to use

| Use INLINE_IMAGE_REAL | Use INLINE_IMAGE_CLAY |
|---|---|
| Named real entity: YCombinator, Stripe, AWS, etc. | Abstract concept: speed, cost savings, architecture |
| Real-world event: demo day, conference, launch party | Metaphorical scene: deploy button, cost pile, building |
| Real-world action: founder at laptop, team in office | Blink-specific product behavior |
| Statistics about real companies / organizations | Creative "before vs after" illustration |
| You want TechCrunch / editorial photo feel | You want branded, fun, consistent Blink visuals |
| Concept has strong, searchable real-world photos | No good real photo exists for this concept |

**When in doubt:** if you can write a specific Google Image search query and expect a high-quality result → use `REAL`. If the concept is too abstract → use `CLAY`.

### Placement Rules

- **Image 1** — after the intro bold answer, before the 1st H2: scope-setter
- **Image 2** — after the most data-heavy section: makes abstract numbers visual
- **Image 3** — after the comparison table or final tool section, before the FAQ

### Comment Formats (copy exactly — the script parses these)

**For real photo search:**
```mdx
<!-- INLINE_IMAGE_REAL: query="Y Combinator demo day founders pitching 2024" | alt="YC Demo Day — founders pitching their startups to investors" -->
```

**For clay character illustration:**
```mdx
<!-- INLINE_IMAGE_CLAY: character surrounded by 6 floating app type screens showing SaaS, portal, marketplace, booking tool, internal tool, membership hub | alt="The range of apps a non-technical founder can build with vibe coding" -->
```

### Run the Script (do not manually process comments)

```bash
node .cursor/skills/ai-seo-articles/scripts/process-inline-images.mjs .todo/seo/drafts/DRAFT_[slug].mdx
```

Exit codes:
- `0` → All images succeeded — proceed to publish
- `2` → No comments found — writer must add INLINE_IMAGE comments first
- `3` → Partial success — check `INLINE_IMAGE_FAILED` markers in draft
- `1` → Fatal error — check MCP server connectivity, bearer token

### Alt Text / Caption Rule

The `alt` value in the comment becomes the **visible caption** below the image.
Write it as one descriptive sentence — what the image shows AND why it's relevant.

- ✅ `"YC Demo Day 2025 — founders pitching to investors in a cohort where 40% used AI to build"`
- ✅ `"The range of apps a non-technical founder can build in days rather than months"`
- ❌ `"startup photo"` / `"illustration of deployment"` / `"Y Combinator"`
